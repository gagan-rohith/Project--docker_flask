AAA Testing

A test is a procedure that exercises a behavior to determine if it works properly. There are various types of tests, such as unit tests, integration tests, and end-to-end tests, but all functional tests perform the same basic function: they try something and report PASS or FAIL.Each test in the pattern focuses on one particular action. The advantage of this focus is that it clearly separates the configuration of the System Under Test (SUT) from the assertions made on it after the action.

AAA stands for Arrange, Act, Assert.

Arrange: The Arrange part is where we set up our environment, i.e. everything we need to run our test code. This includes initializing objects, dependencies, and all data required for the test case.

Act: The act section is where you act upon the system under test. You call one of its methods: pass the dependencies and capture the output value if any.

Assert: The assert section allows you to make the claims about the outcome. This may include the return value, the final state of the SUT and its collaborators, or the methods the SUT called on them.

The main benefit of the AAA pattern is a clear separation of what is being tested from the setup and result verification, which makes the code easier to read and understand. Better readability helps identify key aspects of our tests, thus helping improve the review, maintenance and debugging of our tests.

The AAA pattern is simple and provides a uniform structure for all tests in the suite. This uniform structure is one of its biggest advantages: once you get used to this pattern, you can read and understand the tests more easily. That, in turn, reduces the maintenance cost for your entire test suite.

Here's a basic unit test

def test_abs_for_a_negative_number():
 
  # Arrange
  negative = -5
   
  # Act
  answer = abs(negative)
   
  # Assert
  assert answer == 5

It's natural to begin writing the test with the Arrange section. After all, it comes first, followed by the other two. And in the vast majority of cases, this approach works well. However, starting with the Assert section is also an option. If you use Test-Driven Development, which means writing a test before the production code, you may not know enough about a given feature's behavior, which can result in the creation of a faulty test.
